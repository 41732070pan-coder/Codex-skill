# Transparent Textures Provider Manifest

This manifest lists the `transparent_textures` shared surface provider for `my-design-style`. It now ships **42 distinct seamless textures** across 6 categories. The vocabulary is inspired by Transparent Textures (`https://www.transparenttextures.com/`); each token ships as a **self-contained SVG** — either a base64-inlined seamless PNG tile or a procedural `feTurbulence` grain. No token has a network or external-binary dependency, so all render identically offline, behind a proxy, and in headless SVG-to-bitmap export.

## Usage Rules

- Use through a concrete style's `SurfaceTexturePolicy`; the base workflow consumes provider handles and the style's `allowedTokens`, not the raw folder.
- Each wrapper is a 1920x1080 tiling overlay built on a seamless `<pattern>`. Drop it behind content as a background `<image>`/CSS layer, or inline its `<pattern>` and paint a `<rect>`, then control strength with the layer's opacity.
- Keep opacity within the active style's `opacityRange`; `defaultOpacity` per token in `texture_index.json` is a sensible starting band.
- Place textures below content as background, panel, edge-band, or cover-wash substrate only. Never behind dense body text, data labels, or identity marks.
- Disable texture (`texture-off`) when contrast or data readability would be harmed.
- A style exposes only the tokens that fit its identity. To use a texture in a style that does not list it, add the token to that style's `allowedTokens` first.
- `paper-fibers` is a procedural fallback that renders even if every inlined tile is stripped; keep at least one procedural token available per style.
